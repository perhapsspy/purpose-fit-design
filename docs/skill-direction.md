# 스킬 방향

## 목적

- 기존 구조나 첫 해법이 가능한 방향을 너무 일찍 좁히지 않도록 사용자와 도메인의 목적, 명시된 제약, 현재 근거와 검증 가능한 성공 조건을 먼저 살핀다.

## 유지할 것

- 기존 구현은 기본 권위가 아니라 분류해야 할 증거로 둔다.
- 사용자가 제외한 개념이 다른 계층의 이름으로 되돌아오는 일을 강하게 막는다.
- 맞지 않는 재사용과 “일단 복사하고 나중에 정리”하려는 압력을 막되, 실제 기준을 소유한 기존 컴포넌트나 계약은 보존한다.
- 작은 작업에서는 한 문단 판단으로 충분하게 유지한다.
- 의미 있는 선택에서만 신뢰할 만한 대조안이나 가장 작은 확인을 찾고, 절차를 위해 대안을 만들지 않는다.
- 탐색에서 가능성을 발견했다는 이유만으로 실제 구현 범위를 넓히지 않는다.
- 설계 방향이 정해진 뒤에는 구조화, 구현, 검증 흐름으로 자연스럽게 넘긴다.

## 피할 것

- 스킬을 전면 프로세스 프레임워크로 키우는 방향.
- 모든 재사용을 의심하거나 모든 기존 구현을 거부하는 방향.
- 기준 소유자 확인, 의미 경계 설계와 구현 계획을 이 스킬 안에서 전부 수행하는 방향.
- 예시와 실험 기록을 실행 시 읽는 `SKILL.md`에 계속 추가하는 방향.

## 수정 기준

- 실제 작업에서 누락, 과잉 호출, 늦은 이관이 반복될 때만 배포된 스킬을 수정한다.
- 검증되지 않은 안정성 주장이나 운영 환경에서 검증됐다는 표현은 넣지 않는다.
- 한국어 문서는 영문 기본 스킬의 의미를 보존하되 자연스럽고 직접적으로 쓴다.
- 저장소 운영 문서는 스킬 사용 계약을 대신하지 않는다.
