# 스킬: 목적 적합 설계 (한국어 페어)

> 영문 기본 문서: `SKILL.md`

## 목적

설계나 구현 계획을 시작하기 전에 그 방향이 사용자와 도메인의 목적에 맞는지 확인한다. 기존 구현, 복사하기 쉬운 동작, 대체 처리, 기본값 로직은 설계의 기본 권위가 아니라 평가해야 할 증거로 본다.

끝낼 때는 좁은 방향으로 진행하거나, 판단을 막는 사실 하나를 질문/조사하거나, 전문 작업 흐름으로 넘긴다.

전문 작업이 이미 명확하면 이 스킬 안에서 기준 소유권을 증명하거나 의미 소유자를 배정하거나 코드 구조를 잡지 않는다.

## 적합성 확인

결정에 영향을 주는 내용만 답한다. 작은 작업에서는 짧은 한 문단으로 충분하다.

1. **목적과 성공 기준:** 사용자와 도메인이 얻어야 할 결과, 그리고 가장 작은 관찰 가능한 성공 조건을 말한다.
2. **제약:** 명시적인 사용자 정정, 거절된 개념, "X를 추가하지 말라" 같은 규칙을 고정한다.
3. **증거:** 현재 구현이나 선례를 `기준 소유자`, `파생 호출부`, `레거시/호환`, `증거 후보`, `알 수 없음` 중 하나로 분류한다.
4. **적합성 위험:** 재사용, 대체 처리, 스키마, 어댑터, 지름길, "일단 복사하고 나중에 정리" 중 목적에서 벗어나게 만들 수 있는 경로를 말한다.
5. **방향 또는 넘김:** 좁은 목적 적합 방향을 말하거나 하나의 전문 작업 흐름을 지정한다.
6. **검증:** 선택한 방향이 틀렸음을 드러낼 수 있는 가장 작은 확인을 말한다.

목적, 제약, 증거 상태, 성공 기준 중 설계에 영향을 주는 것이 불명확하면 하나를 조사하거나 집중 질문을 한다. 현재 구현 형태로 빈칸을 채우지 않는다.

명시적인 사용자 정정은 반드시 지켜야 할 제약으로 다룬다. 사용자가 다시 검토하자고 하지 않는 한 거절된 필드, 상태, 화면 패턴, 스키마, 제한 시간, 정책, 대체 처리를 다른 계층 이름으로 다시 들여오지 않는다.

낮은 계층의 대체 처리는 명령 의미, 표시 상태, 재시도, 감사 기록, 저장, 테스트, 정책, 다른 호출부에 영향을 줄 수 있으면 설계 결정으로 본다. 거절된 제약을 되살리거나 의미에 영향을 주는 동작의 소유자가 없으면 멈추고 의미 경계 작업으로 넘긴다.

경로, 화면, 어댑터, 실시간 처리, 표현 계층에서 도메인/상태 정체성을 대체 처리 사슬로 계산한다면 단순한 범위 조절 문제로 보지 않는다. 현재 소유자가 명명되어 있지 않으면 편집 계획 전에 기준 소유자 확인이나 의미 경계 확인으로 넘긴다.

전송 감시 장치나 인프라 보호 장치는 전송/인프라 계층이 소유하고, 연결 생존성이나 자원 보호에 한정되고, 도메인 필드/상태/정책/기본값을 만들지 않고, 사용자에게 확정 표시되는 상태를 바꾸지 않고, 호출부에 업무 의미로 전달될 수 없을 때만 의미 경계 작업 없이 진행할 수 있다.

긴급함은 적합성 증거가 아니다. 사용자가 지름길을 요청하면 지금 목적, 제약, 소유자 경계를 보존하는 가장 작은 범위를 제안한다. "일단 복사하고 나중에 정리"를 설계 전략으로 제시하지 않는다. 임시 구현도 코드 시작 전에 명시적인 어댑터, 감싸기 계층, 되돌림 경계가 필요하다.

## 전문 작업 흐름으로 넘기기

전문 작업의 발동 조건이 이미 명확하면 이 스킬을 건너뛰고 그 작업을 직접 사용한다. 그렇지 않으면 이 확인 뒤에:

- 따를 기준이 증명되지 않았으면 기준 소유자 확인을 사용한다.
- 하나의 의미가 계층 사이에서 엇나갈 수 있으면 의미 경계 설계를 사용한다.
- 선택지, 필드, 추상화, 후속 수정이 넓어질 수 있거나 소유자와 설계 방향이 정해진 뒤 코드 모양이 다음 문제라면 구조 중심 변경 경계 계획을 사용한다.
- 사용자 의도, 표현, 비동기 작업, 최신성 판단이 섞일 수 있으면 상호작용 상태 흐름 분석을 사용한다.

## 위험 신호

- 현재 목적에 맞는지 확인하지 않고 "이 컴포넌트가 이미 있으니 재사용"하는 경우.
- 사용자 목표나 소유자 증거 없이 "보안/일관성상 스키마/목록/제한 시간이 필요"하다고 말하는 경우.
- 거절된 필드, 상태, 화면 패턴, 스키마, 제한 시간, 정책, 대체 처리를 낮은 계층의 구현 세부사항으로 다시 들여오는 경우.
- 다른 도메인의 상태, 배치, 정책, 생명주기를 소유한 표면을 "일단 복사하고 나중에 정리"로 받아들이는 경우.
- 화면, 경로, 어댑터, 실시간 코드가 `x || y || z` 대체 처리 사슬로 도메인 의미를 결정하는 경우.
- 호출부 경계의 동작을 보호하지 않고 도우미 내부 테스트만 증명하는 경우.
- 설계 설명 대부분이 현재 구현이 괜찮은 이유를 설명하는 경우.
- 사용자가 설계, 소유권, 목적, 적합성을 다시 보자고 했는데 곧바로 편집으로 이동하는 경우.

## 최종 확인

구현이나 최종 추천 전에 확인한다:

- 목적이 단지 "현재 구현을 작동하게 만들기"가 아니다.
- 선택한 방향이 사용자/도메인 목적과 관찰 가능한 성공 조건에 맞다.
- 명시적인 사용자 정정이나 거절된 개념을 다시 들여오지 않았다.
- 긴급함이 제약, 소유자 경계, 되돌림 경계, 검증을 덮지 않았다.
- 기존 코드는 편의가 아니라 증거 상태로 분류했다.
- 의미 결정에는 하나의 소유자가 있거나, 모르면 모른다고 명명했다.
- 절차의 무게가 작업 크기에 맞다.
