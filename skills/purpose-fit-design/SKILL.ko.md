# 스킬: 목적에 맞는 설계

> 영문 기본 문서: `SKILL.md`

## 목적

설계나 구현 계획에 들어가기 전에 현재 구현의 모양을 답으로 삼지 말고, 사용자와 도메인이 원하는 결과와 가장 작게 관찰할 수 있는 성공 조건을 먼저 적는다. 그 성공 조건, 명시된 제약과 확인된 소유자 경계가 요구하는 책임과 관계에서 출발한다. 기존 구현, 복사하기 쉬운 동작, 대체 처리, 기본값 계산과 앞으로의 가능성은 설계의 기본형이 아니라 판단 자료로 본다.

끝낼 때는 잠정 방향, 판단에 도움이 되는 가장 작은 확인 방법, 또는 선택을 실제로 막는 전문 질문 하나를 남긴다. 더 넓은 가능성은 자유롭게 살피고 제안할 수 있지만, 가능성을 찾았다는 이유만으로 구현 범위를 넓히지는 않는다.

전문 작업이 이미 명확하면 이 스킬 안에서 기준 소유권을 증명하거나 의미 소유자를 배정하거나 코드 구조를 잡지 않는다.

## 초기 방향 살피기

결정에 영향을 주는 내용만 답한다. 작은 작업에서는 짧은 한 문단으로 충분하다.

1. **기준 잡기:** 사용자와 도메인이 얻어야 할 결과, 가장 작게 관찰할 수 있는 성공, 명시적인 정정이나 제외한 개념, 선택에 영향을 주는 현재 근거와 아직 모르는 점을 말한다.
2. **가능성 살피기:** 기존 구조, 선례, 대체 처리·기본값 계산, 지름길 압력, 막연한 보안·일관성 주장이 방향을 선점하는지 본다. 영향이 큰 선택이 있을 때만 비교할 만한 다른 방향 하나나 판단을 가를 가장 작은 확인 방법을 찾는다.
3. **방향 정하기:** 잠정 방향과 그 방향을 바꿀 근거를 말하거나, 가장 작은 확인 방법을 정하거나, 후보 비교를 막는 구체적인 전문 질문 하나를 넘긴다.

목적, 제약, 증거 상태, 성공 기준 중 설계에 영향을 주는 것이 불명확하면 하나를 조사하거나 집중 질문을 한다. 현재 구현 형태로 빈칸을 채우지 않는다.

현재 구현이나 선례의 역할이 선택에 영향을 주면 `기준 소유자`, `파생 호출부`, `기존 호환 경로`, `근거 후보`, `알 수 없음` 중 하나로 분류한다.

절차를 채우기 위해 대안을 만들지 않는다. 영향이 큰 선택은 중요한 책임, 영속 상태, 정책, 의존성, 생명주기, 공개 계약, 호환성이나 되돌리기 어려운 결합을 추가하거나 바꾸는 선택이다.

명시적인 사용자 정정은 방향과 범위를 구속하는 제약으로 다룬다. 정정과 주변 요청에서 영향 범위를 판단해, 국소 정정을 재설계로 넓히거나 광범위한 정정을 국소 수정으로 줄이지 않는다. 영향받지 않은 목적, 제약과 이미 확인된 소유자 경계는 보존한다. 사용자가 다시 검토하자고 하지 않는 한 제외한 필드, 상태, 화면 패턴, 스키마, 제한 시간, 정책, 대체 처리를 다른 계층의 이름으로 다시 들여오지 않는다.

낮은 계층의 대체 처리는 명령 의미, 표시 상태, 재시도, 감사 기록, 저장, 테스트, 정책, 다른 호출부에 영향을 줄 수 있으면 설계 결정으로 본다. 거절된 제약을 되살리면 들여오지 않는다. 의미 소유권의 미확정이 현재 방향을 바꾸거나 후보 비교를 막는다면 그 구체적인 질문을 의미 경계 작업으로 넘긴다.

경로, 화면, 어댑터, 실시간 처리 또는 표현 계층이 대체 처리 사슬로 도메인 의미나 상태의 정체성을 결정한다면, 이를 단순한 범위 조절 문제로 보지 않는다. 소유자가 불분명한 상태가 현재 동작이나 설계 선택에 영향을 줄 때는 편집 계획 전에 기준 소유자 확인이나 의미 경계 확인을 사용한다. 소유자가 아직 이름 붙지 않았다는 이유만으로 전문 스킬을 연달아 호출하지 않는다.

전송 감시 장치나 인프라 보호 장치는 전송/인프라 계층이 소유하고, 연결 생존성이나 자원 보호에 한정되고, 도메인 필드/상태/정책/기본값을 만들지 않고, 사용자에게 확정 표시되는 상태를 바꾸지 않고, 호출부가 이를 업무 의미로 해석할 수 없을 때만 의미 경계 작업 없이 진행할 수 있다.

긴급함은 적합성의 근거가 아니다. 사용자가 지름길을 요청하면 지금 목적, 제약, 소유자 경계를 보존하는 가장 작은 범위를 제안한다. "일단 복사하고 나중에 정리"를 설계 전략으로 제시하지 않는다. 임시 구현도 어떻게 되돌릴지 분명해야 한다. 현재 결합 때문에 되돌리거나 격리하기가 실제로 어려울 때만 어댑터, 감싸기 계층이나 되돌림 경계를 추가한다.

## 전문 스킬로 넘기기

전문 스킬의 적용 조건이 이미 명확하면 이 스킬을 건너뛰고 해당 스킬을 바로 사용한다. 그렇지 않으면 이 확인 뒤에:

- 따라야 할 기준이 어디에 있는지 확인되지 않았으면 기준 소유자 확인을 사용한다.
- 하나의 의미가 계층 사이에서 엇나갈 수 있으면 의미 경계 설계를 사용한다.
- 방향이 정해진 뒤 관련 동작이나 판단 소유를 추적하기 어렵거나, 부작용이 검증을 가리거나, 경계가 늘어나거나, 제안된 구조가 복잡도를 이동시킬 수 있으면 구조 중심 탐색을 사용한다.
- 사용자 의도, 표현, 비동기 작업, 최신성 판단이 섞일 수 있으면 상호작용 상태 흐름 분석을 사용한다.

소유자가 아직 이름 붙지 않았거나 넓은 위험 신호가 있다는 이유만으로 전문 스킬을 연달아 호출하지 않는다. 풀리지 않은 전문 질문이 현재 의미나 방향을 바꾸거나, 가능한 후보의 비교를 막을 때만 넘긴다.

## 위험 신호

- 현재 목적에 맞는지 확인하지 않고 "이 컴포넌트가 이미 있으니 재사용"하는 경우.
- 사용자 목표나 소유자 근거 없이 "보안이나 일관성을 위해 스키마, 목록, 제한 시간이 필요하다"고 말하는 경우.
- 제외한 필드, 상태, 화면 패턴, 스키마, 제한 시간, 정책, 대체 처리를 낮은 계층의 구현 세부사항으로 다시 들여오는 경우.
- 복사하려는 대상이 다른 도메인의 상태, 화면 배치, 정책 또는 생명주기를 맡고 있는데도 "일단 복사하고 나중에 정리"로 받아들이는 경우.
- 화면, 경로, 어댑터, 실시간 코드가 `x || y || z` 대체 처리 사슬로 도메인 의미를 결정하는 경우.
- 호출부 경계의 동작을 보호하지 않고 도우미 내부 테스트만 증명하는 경우.
- 설계 설명 대부분이 현재 구현이 괜찮은 이유를 설명하는 경우.
- 사용자가 설계, 소유권, 목적, 적합성을 다시 보자고 했는데 곧바로 편집으로 이동하는 경우.

## 최종 확인

구현이나 최종 추천 전에 확인한다:

- 목적이 단지 "현재 구현을 작동하게 만들기"가 아니다.
- 선택한 방향이 사용자/도메인 목적과 관찰 가능한 성공 조건에 맞다.
- 명시적인 사용자 정정이나 거절된 개념을 다시 들여오지 않았다.
- 각 정정의 도달 범위가 요청과 맞고, 영향받지 않은 목적, 제약과 이미 확인된 소유자 경계가 유지되었다.
- 긴급함이 제약, 소유자 경계, 신뢰할 만한 되돌림 설명, 검증을 덮지 않았다.
- 기존 코드는 편의가 아니라 증거 상태로 분류했다.
- 의미를 정하는 경로는 서로 경쟁하지 않아야 한다. 아직 정해지지 않았다면 그 사실을 명시했다. 이 경로는 한 소유자일 수도 있고, 정해진 조합 순서나 조정·충돌 해결 규약일 수도 있다.
- 절차의 무게가 작업 크기에 맞다.
