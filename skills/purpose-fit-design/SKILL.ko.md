# 스킬: 목적에 맞는 설계

> 영문 기본 문서: `SKILL.md`
> 설명 동기화: 사용자·도메인 목적, 명시 제약, 현재 근거와 확인 가능한 성공 조건에 맞춰 초기 설계나 구현 계획의 방향을 정할 때 씁니다. 새 기능, 기존 구현 재사용과 임시 구현의 방향을 코딩 전에 판단하며, 방향이 이미 정해졌거나 전문 문제가 분명한 경우에는 쓰지 않습니다.

## 목적

사용자·도메인 목적, 명시 제약, 현재 근거와 관찰 가능한 최소 성공 조건에 맞는 초기 방향을 정한다. 기존 구현과 선례는 자동 기준이 아니라 판단 근거로 다룬다.

잠정 방향, 판단에 도움이 되는 가장 작은 확인, 또는 선택을 막는 질문 하나로 끝낸다. 가능성을 더 찾았다는 이유만으로 구현 범위를 넓히지 않는다.

## 적합성 확인

결정에 영향을 주는 내용만 답한다. 작은 작업은 한 문단이면 충분하다.

1. **목적:** 사용자·도메인이 얻어야 할 결과와 관찰 가능한 최소 성공 조건을 적는다.
2. **제약:** 명시적인 정정, 제외한 개념과 선택의 한계를 적는다.
3. **근거:** 기존 코드, 선례와 기본값이 실제로 증명하는 것과 아직 모르는 것을 구분한다. 긴급함은 전달 제약이지 적합성의 근거가 아니다.
4. **방향:** 잠정 방향과 그 방향을 바꿀 근거 또는 가장 작은 확인 방법을 적는다.

선택에 영향을 주는 사실이 빠졌다면, 그 사실을 조사하거나 질문 하나로 확인한다. 현재 구현으로 빈칸을 채우지 않는다.

중요한 책임, 영속 상태, 정책, 의존성, 생명주기, 공개 계약, 호환성이나 되돌리기 어려운 결합이 달라질 때만 다른 방향을 비교한다.

명시적인 사용자 정정은 구속력 있게 다룬다. 요청에 맞는 범위로 적용하고, 영향받지 않은 제약을 보존하며, 제외한 개념을 다른 이름으로 다시 들여오지 않는다.

기술적 기본값이나 대체 처리, 보안·일관성에 관한 주장도 사용자에게 보이는 동작, 도메인 의미, 정책을 바꾸면 설계 선택으로 다룬다. 그렇지 않으면 기술 세부사항으로 둔다.

임시 구현은 목적과 제약을 지키는 가장 작은 범위로 고른다. 검증하고 되돌릴 방법은 분명히 하되, 실제로 도움이 될 때만 어댑터나 별도 경계를 추가한다.

## 전문 스킬로 넘기기

전문 문제가 이미 분명하면 이 스킬을 건너뛴다. 그렇지 않으면 풀리지 않은 사실이 방향을 막을 때만 넘긴다.

- 따라야 할 계약을 모르면 기준 소유자를 확인한다.
- 하나의 의미가 계층마다 달라질 수 있으면 의미 경계를 확인한다.
- 사용자 의도, 비동기 작업, 표현, 최신성 판단이 섞이면 상호작용 상태 흐름을 확인한다.
- 방향이 정해진 뒤 코드 구조나 검증만 남았다면 `structure-first`를 사용한다.

## 위험 신호

- 목적과 제약을 확인하지 않고 이미 있다는 이유만으로 재사용하는 경우.
- 막연한 보안, 일관성이나 미래 가능성을 근거로 정책이나 구조를 추가하는 경우.
- 명시적으로 제외한 개념을 다른 이름이나 계층으로 다시 들여오는 경우.
- 검증과 되돌림 설명 없이 "일단 복사하고 나중에 정리"하는 경우.

## 최종 확인

구현이나 최종 추천 전에 확인한다:

- 방향이 목적, 제약과 관찰 가능한 성공에 맞다.
- 기존 코드와 선례를 자동 기준이 아니라 근거로 다뤘다.
- 명시적인 정정과 제외한 개념을 지켰다.
- 선택을 확인할 방법이나 아직 모르는 사실이 분명하다.
- 절차의 무게가 선택의 크기와 되돌리기 어려운 정도에 맞다.
