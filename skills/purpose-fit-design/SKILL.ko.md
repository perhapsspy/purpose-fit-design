# 스킬: 목적 적합 설계 (한국어 페어)

> 영문 기본 문서: `SKILL.md`

## 목적

설계나 구현 계획을 시작하기 전에 현재 구현 형태를 물려받지 않고 사용자·도메인 목적과 가장 작은 관찰 가능한 성공을 표현한다. 그 성공, 명시 제약과 확인된 소유자 경계가 요구하는 책임과 관계에서 출발한다. 기존 구현, 복사하기 쉬운 동작, 대체 처리, 기본값 로직과 미래 가능성은 설계의 기본 형태가 아니라 평가할 증거로 본다.

끝낼 때는 잠정 방향, 가장 작은 유용한 확인, 또는 선택을 실제로 막는 전문 질문 하나를 남긴다. 더 넓은 가능성은 자유롭게 탐색하고 제안할 수 있지만, 발견만으로 구현 범위를 넓힐 권한이 생기지는 않는다.

전문 작업이 이미 명확하면 이 스킬 안에서 기준 소유권을 증명하거나 의미 소유자를 배정하거나 코드 구조를 잡지 않는다.

## 방향 열기

결정에 영향을 주는 내용만 답한다. 작은 작업에서는 짧은 한 문단으로 충분하다.

1. **기반:** 사용자·도메인 결과, 가장 작은 관찰 가능한 성공, 명시적인 정정이나 거절된 개념, 선택에 영향을 주는 현재 증거와 미확정을 말한다.
2. **열기:** 기존 구조, 선례, 대체 처리·기본값 로직, 지름길 압력, 추상적인 보안·일관성 주장이 방향을 선점하는지 본다. 의미 있는 선택이 있을 때만 신뢰할 만한 대조안 하나나 방향을 가를 가장 작은 확인을 찾는다.
3. **선택:** 잠정 방향과 그 방향을 바꿀 증거를 말하거나, 가장 작은 유용한 확인을 정하거나, 후보 비교를 막는 구체적인 전문 질문 하나를 넘긴다.

목적, 제약, 증거 상태, 성공 기준 중 설계에 영향을 주는 것이 불명확하면 하나를 조사하거나 집중 질문을 한다. 현재 구현 형태로 빈칸을 채우지 않는다.

현재 구현이나 선례의 역할이 선택에 영향을 주면 `기준 소유자`, `파생 호출부`, `레거시/호환`, `증거 후보`, `알 수 없음` 중 하나로 분류한다.

절차를 수행하기 위해 대안을 만들지 않는다. 의미 있는 선택은 중요한 책임, 영속 상태, 정책, 의존성, 생명주기, 공개 계약, 호환성이나 되돌리기 어려운 결합을 추가하거나 바꾸는 선택이다.

명시적인 사용자 정정은 방향과 범위를 구속하는 제약으로 다룬다. 정정과 주변 요청에서 도달 범위를 판단해, 국소 정정을 재설계로 넓히거나 광범위한 정정을 국소 수정으로 줄이지 않는다. 영향받지 않은 목적, 제약과 이미 확인된 소유자 경계는 보존한다. 사용자가 다시 검토하자고 하지 않는 한 거절된 필드, 상태, 화면 패턴, 스키마, 제한 시간, 정책, 대체 처리를 다른 계층 이름으로 다시 들여오지 않는다.

낮은 계층의 대체 처리는 명령 의미, 표시 상태, 재시도, 감사 기록, 저장, 테스트, 정책, 다른 호출부에 영향을 줄 수 있으면 설계 결정으로 본다. 거절된 제약을 되살리면 들여오지 않는다. 의미 소유권의 미확정이 현재 방향을 바꾸거나 후보 비교를 막는다면 그 구체적인 질문을 의미 경계 작업으로 넘긴다.

경로, 화면, 어댑터, 실시간 처리, 표현 계층에서 도메인/상태 정체성을 대체 처리 사슬로 계산한다면 단순한 범위 조절 문제로 보지 않는다. 소유권 미확정이 현재 동작이나 설계 선택에 영향을 줄 때는 편집 계획 전에 기준 소유자 확인이나 의미 경계 확인을 사용한다. 소유자가 아직 명명되지 않았다는 사실만으로 전문 작업 사슬을 시작하지 않는다.

전송 감시 장치나 인프라 보호 장치는 전송/인프라 계층이 소유하고, 연결 생존성이나 자원 보호에 한정되고, 도메인 필드/상태/정책/기본값을 만들지 않고, 사용자에게 확정 표시되는 상태를 바꾸지 않고, 호출부에 업무 의미로 전달될 수 없을 때만 의미 경계 작업 없이 진행할 수 있다.

긴급함은 적합성 증거가 아니다. 사용자가 지름길을 요청하면 지금 목적, 제약, 소유자 경계를 보존하는 가장 작은 범위를 제안한다. "일단 복사하고 나중에 정리"를 설계 전략으로 제시하지 않는다. 임시 구현에는 신뢰할 만한 되돌림 설명이 필요하다. 현재 결합이 되돌림이나 격리를 실질적으로 어렵게 할 때만 어댑터, 감싸기 계층이나 되돌림 경계를 추가한다.

## 전문 작업 흐름으로 넘기기

전문 작업의 발동 조건이 이미 명확하면 이 스킬을 건너뛰고 그 작업을 직접 사용한다. 그렇지 않으면 이 확인 뒤에:

- 따를 기준이 증명되지 않았으면 기준 소유자 확인을 사용한다.
- 하나의 의미가 계층 사이에서 엇나갈 수 있으면 의미 경계 설계를 사용한다.
- 방향이 정해진 뒤 관련 동작이나 판단 소유를 추적하기 어렵거나, 부작용이 검증을 가리거나, 경계가 늘어나거나, 제안된 구조가 복잡도를 이동시킬 수 있으면 구조 중심 탐색을 사용한다.
- 사용자 의도, 표현, 비동기 작업, 최신성 판단이 섞일 수 있으면 상호작용 상태 흐름 분석을 사용한다.

소유자가 아직 명명되지 않았거나 넓은 위험 신호가 있다는 이유만으로 전문 작업 사슬을 시작하지 않는다. 미확정된 전문 질문이 현재 의미나 방향을 바꾸거나, 가능한 후보의 비교를 막을 때만 넘긴다.

## 위험 신호

- 현재 목적에 맞는지 확인하지 않고 "이 컴포넌트가 이미 있으니 재사용"하는 경우.
- 사용자 목표나 소유자 증거 없이 "보안/일관성상 스키마/목록/제한 시간이 필요"하다고 말하는 경우.
- 거절된 필드, 상태, 화면 패턴, 스키마, 제한 시간, 정책, 대체 처리를 낮은 계층의 구현 세부사항으로 다시 들여오는 경우.
- 다른 도메인의 상태, 배치, 정책, 생명주기를 소유한 표면을 "일단 복사하고 나중에 정리"로 받아들이는 경우.
- 화면, 경로, 어댑터, 실시간 코드가 `x || y || z` 대체 처리 사슬로 도메인 의미를 결정하는 경우.
- 호출부 경계의 동작을 보호하지 않고 도우미 내부 테스트만 증명하는 경우.
- 설계 설명 대부분이 현재 구현이 괜찮은 이유를 설명하는 경우.
- 사용자가 설계, 소유권, 목적, 적합성을 다시 보자고 했는데 곧바로 편집으로 이동하는 경우.

## 최종 확인

구현이나 최종 추천 전에 확인한다:

- 목적이 단지 "현재 구현을 작동하게 만들기"가 아니다.
- 선택한 방향이 사용자/도메인 목적과 관찰 가능한 성공 조건에 맞다.
- 명시적인 사용자 정정이나 거절된 개념을 다시 들여오지 않았다.
- 각 정정의 도달 범위가 요청과 맞고, 영향받지 않은 목적, 제약과 이미 확인된 소유자 경계가 유지되었다.
- 긴급함이 제약, 소유자 경계, 신뢰할 만한 되돌림 설명, 검증을 덮지 않았다.
- 기존 코드는 편의가 아니라 증거 상태로 분류했다.
- 의미 결정에는 비경쟁적인 resolution path가 있거나, 모르면 모른다고 명명했다. 그 경로는 단일 소유자, 명시적인 조합 순서, 조정·충돌 해결 protocol일 수 있다.
- 절차의 무게가 작업 크기에 맞다.
